# A3 - Architecture and Dependencies

## Package Layout
- `src/resonance_risk_screening/`
  - `io.py` data loading and schema checks
  - `preprocessing.py` alignment, interpolation, outlier handling
  - `proxies.py` physics-guided proxy computations
  - `clustering.py` PAM state reconstruction
  - `risk_model.py` susceptibility score + ordinal model
  - `benchmarks.py` baseline comparators
  - `validation.py` temporal CV and robustness analysis
  - `artifacts.py` manuscript table/figure generation
  - `pipeline.py` orchestration and interactive workflow
  - `cli.py` CLI entrypoints
  - `logging_config.py` structured logging

## Architectural Pattern
- Composition-first architecture:
  - `ResonancePipeline` composes stage modules.
  - Each stage has explicit input/output data contracts.
- Design patterns:
  - Strategy-like stage decomposition (ingest, preprocess, proxy, cluster, model, validate, artifacts).
  - Adapter pattern (`OrdinalRiskModel`) for unified probability output contract.
  - Fluent interface (`ScreeningSession`) for method-chaining user workflows.
- Abstract interfaces:
  - `BasePipeline` and `BaseModelAdapter` define extension contracts.
- Deterministic execution:
  - random seeds configured centrally;
  - temporal split constraints enforced in validation layer.

## Data and Logging Flow
1. `io` -> raw operational dataframe.
2. `preprocessing` -> cleaned aligned dataframe.
3. `proxies` -> feature matrix and stiffness metrics.
4. `clustering` -> state labels and medoids.
5. `risk_model` -> score, labels, probabilities.
6. `validation`/`benchmarks` -> evaluation outputs.
7. `artifacts` -> manuscript tables/figures.
8. `logging_config` writes run and stats events to `logs/`.

## Pipeline Architecture Notes
- Interactive pipeline wraps the same core modules used by API.
- CLI wizard avoids hidden logic by persisting resolved configuration.
- Stage-level failures are surfaced with user-readable context.
