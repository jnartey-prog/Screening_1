# A6 - User Experience and Non-Coder Accessibility

## Target Users
- Primary: utility analysts and power-quality researchers with low-to-moderate coding fluency.
- Secondary: developers extending algorithms and validation modules.

## Accessibility Design
- Guided CLI workflow (`resonance-screen`) with sensible defaults.
- Single-command pipeline (`resonance-pipeline`) for full analysis.
- Short top-level API: `read()`, `screen()`, `ScreeningSession().load(...).prep().fit().summary()`.
- User-facing errors include missing-column hints and corrective actions.

## Progressive Usage Levels
1. No/low-code: CLI prompts for input/output and runs full flow.
2. Scripted: `ResonancePipeline().run(input_path, output_dir)`.
3. Advanced: direct use of module-level APIs (`compute_proxies`, `train_ordinal_model`, etc.).

## Non-Coder Validation Targets
- One-liner workflow works on fixture dataset.
- Troubleshooting FAQ and glossary included in docs.
- Artifact manifest generation confirms output completeness.
- Template script with clear "edit here" sections is provided for first-time users.
